## 0. Executive Summary

**한 문장 비전:** 도쿄의 집 부엌에서, 근처 슈퍼에서 산 재료만으로 실패 없이 한 끼를 완성하게 한다.

도쿄 거주 한국인이 집밥을 하려 할 때의 3대 페인포인트 — ① 한국 레시피가 전제하는 재료가 일본 슈퍼엔 없거나 이름이 다름, ② 인분 조절 시 분량 계산이 번거롭고 자주 틀림, ③ 광고·잡담 섞인 블로그/영상을 젖은 손으로 보기 힘듦 — 을 **재료(일본 슈퍼 한정·한/일 병기) → 양(인분 자동 계량) → 조리(모바일 단계 뷰)** 흐름으로 한 번에 해결한다.

**핵심 차별점 3가지**
1. **인분 자동 계량** — 인분만 입력하면 물 ml·간장 큰술·소금 g까지 재료별 특성에 맞게 자동 재계산.
2. **일본 슈퍼 재료 한정 + 한/일 병기** — 도쿄 슈퍼에서 실제로 살 수 있는 재료로만 구성, 매대에서 바로 찾도록 일본어명 병기.
3. **모바일 조리 전용 뷰** — 큰 글씨, 한 단계씩, 화면 꺼짐 방지, 스텝 체크.

**기술 방향:** 백엔드 없는 **바닐라 HTML+CSS+JS(ES 모듈) 정적 웹앱**. 빌드 없이 `git push`로 GitHub Pages 배포. 레시피는 **카테고리별 JSON 파일**로 관리해 코드 수정 없이 추가 가능. 앱의 복잡성은 UI가 아니라 **인분 계산 엔진**에 있으며, 이는 순수 함수 모듈로 분리해 테스트로 방어한다.

**MVP 범위:** 카테고리 탐색 → 레시피 상세(재료 자동 계량 + 조리 가이드) → 모바일 반응형 → GitHub Pages 배포 + 시드 레시피 10~15개. "고르고 → 인분 정하면 양이 딱 나오고 → 따라 만들면 완성"이 되면 MVP 완료.

## 1. 종합 결정사항 (Single Source of Truth)

> 5개 팀 산출물을 통합하며 확정한 **공통 기준**. 이후 모든 섹션은 이 규칙을 따른다. (팀별 초안에서 명칭이 달랐던 부분을 여기서 통일했다.)

### 1.1 기술 스택
- **바닐라 HTML + CSS + JS(ES 모듈)**, 빌드 없음. → 이유·비교표는 §5.1.
- **GitHub Pages** 배포. 자산·데이터는 **상대경로**, **해시 라우팅** 사용(정적 호스팅 궁합).
- PWA(오프라인·홈화면 추가)는 **Phase 2** 선택 기능.

### 1.2 데이터 저장
- **카테고리별 JSON 파일**: `data/recipes/한식.json`, `양식.json`, `중식.json`, `일식.json`. 각 파일 = 레시피 배열. 추가 = 객체 하나 append.
- 파일명·URL에 한글 사용 가능(fetch 시 `encodeURIComponent`).
- 데이터는 **기준 인분(baseServings) 기준값**으로 저장하고, 표시 시점에 배율을 곱해 계산(데이터 일관성).

### 1.3 계량 표준 (일본 기준)
| 단위 | 값 | 비고 |
|---|---|---|
| 큰술(大さじ) | 15 ml | |
| 작은술(小さじ) | 5 ml | |
| 컵(カップ) | **200 ml** | 일본 표준(미국 240ml 아님) |
| 개·모·대·장·쪽 | 정수/개수 | 셀 수 있는 재료 |
| g / ml | 무게·부피 | 내부 계산 기본 단위 |

허용 unit 화이트리스트: `g, ml, 큰술, 작은술, 컵, 개, 모, 대, 장, 쪽, 줌, 적당량`.

### 1.4 `scaleType` — 인분 스케일링 규칙 (⚠ 통일 완료)

배율 `ratio = 목표인분 / baseServings`. 각 재료는 아래 **5종 중 하나**를 가진다.

| scaleType | 계산식 | 적용 대상 | 반올림 |
|---|---|---|---|
| `linear` | `양 × ratio` | 물·간장·기름·설탕·미림, 주재료(고기·밥·채소 g) | 표시 단위로 스냅 |
| `sqrt` | `양 × √ratio` | 소금·후추·마늘·생강·고춧가루·고추장·두반장 등 **강한 간/향** | 표시 단위로 스냅 |
| `count` | `max(1, round(양 × ratio / step) × step)` | 계란·양파·두부(모)·대파(대) 등 **셀 수 있는 것**. `roundStep` 기본 1, 반 단위 허용 시 0.5 | roundStep 단위 |
| `fixed` | `양`(그대로) | 인분과 무관한 고정량(월계수잎 1장, 코팅용 소량) | 없음 |
| `to-taste` | 계산 안 함 | 소금 약간·후추 약간·고명(`amount: null`) | "기호에 따라" 표시 |

> **명칭 매핑(초안 → 확정):** 도메인팀의 `sublinear` = **`sqrt`**, `linear-round` = **`count`(roundStep 사용)**. 프론트/QA팀의 `count`는 그대로. 본 문서 이하 모든 표는 확정 명칭을 쓴다.

**왜 `sqrt`인가:** 간·향을 인분에 정비례로 늘리면 짜거나 매워진다(미각 인지 비선형). 가정요리에서 검증된 절충이 제곱근 비례다. 예) 2인분 소금을 8인분(ratio=4)으로 → linear면 4배(과함), √4=2배가 현실적.

**표시 반올림(스냅):** 계산은 원본 정밀도로 하고 **표시 직전에만** 반올림해 오차 누적을 막는다.
- `ml`·`g`: 5 단위 / `큰술`·`컵`: 0.5 단위 / `작은술`: 0.25~0.5 단위
- 분수 표기: `0.25→¼`, `0.33→⅓`, `0.5→½`, `0.67→⅔`, `0.75→¾`, 정수부는 앞에(`1½`, `2¼`). 매핑 실패 시 소수 1자리 폴백.

**대량 조리 안내:** `ratio ≥ 3`이면 국·찌개 물은 "5~10% 줄이고 끓이며 간 조절" 안내를 자동 노출(증발 비율이 낮아 국물이 남음). 조리시간은 스케일하지 않고 "양이 많으면 익는 시간 약간 증가"만 안내.

### 1.5 레시피 JSON 스키마 (확정)

```jsonc
{
  "id": "kr-miyeokguk-beef",        // 소문자 kebab-case, 파일 내 고유
  "title": { "ko": "소고기미역국", "ja": "牛肉わかめスープ" },
  "category": "한식",                // 한식 | 양식 | 중식 | 일식
  "baseServings": 2,                 // 기준 인분 (ingredients.amount의 기준)
  "time": { "prepMin": 15, "cookMin": 30 },
  "difficulty": "easy",              // easy | normal | hard
  "tags": ["국물", "미역"],
  "ingredients": [
    {
      "name": { "ko": "국간장", "ja": "薄口醤油 / うすくちしょうゆ" },
      "amount": 1,                   // baseServings 기준 수량. to-taste면 null
      "unit": "큰술",                // §1.3 화이트리스트
      "scaleType": "linear",         // §1.4 5종
      "roundStep": 0.5,              // (선택) count에서 반 단위 허용 시
      "group": "양념",               // (선택) 재료 묶음: 양념장·육수·고명 등
      "note": "薄口는 짜니 간 보며",   // (선택) 대체·손질 팁
      "optional": false              // (선택)
    }
  ],
  "steps": ["...", "..."],           // 조리 순서(스케일 무관 텍스트). {order,text,timeMin} 형태 확장 가능
  "tips": ["..."]                    // (선택)
}
```

### 1.6 열린 이슈 (개발 착수 전 확인 필요)
- **인분 범위**: UX 초안은 1~4인분(스테퍼), QA는 0.5~대량까지 방어. → **UI 스테퍼는 1~4로 고정, 계산 엔진은 0.5~임의값 방어**로 결정. (URL `?n=`로 임의값 진입 시에도 안전).
- **steps 자료형**: MVP는 `string[]`로 시작, 단계 타이머(P1) 도입 시 `{order,text,timeMin}` 객체로 확장.
- **to-taste 렌더링**: 엔진은 `to-taste`(amount:null) 및 `fixed`+amount:0 둘 다 "기호에 따라/적당량"으로 안전 처리.

## 2. 제품 기획

### 2.1 제품 비전 & 문제 정의

**비전 한 문장:** 도쿄의 집 부엌에서, 근처 슈퍼에서 산 재료만으로 실패 없이 한 끼를 완성하게 한다.

도쿄에 사는 한국인이 집밥을 해 먹으려 할 때 겪는 페인포인트는 명확하다.

- **재료 미스매치:** 한국 레시피 블로그·유튜브는 "다시다 1스푼", "청양고추" 같은 한국 마트 기준 재료를 전제한다. 일본 슈퍼에는 없거나 이름이 달라서, 매번 대체재를 검색하느라 요리 시작 전에 지친다.
- **양 계산의 번거로움:** 대부분 레시피가 "2~3인분" 같은 고정 분량이다. 혼자 사는 1인분, 손님 온 4인분으로 바꾸려면 머릿속으로 비율 계산을 해야 하고, "간장 1.5큰술의 2/3는 몇 큰술?" 같은 데서 자주 틀린다.
- **조리 중 화면 사용성:** 광고·잡담·스크롤이 뒤섞인 블로그/영상을 젖은 손으로 폰에서 넘겨보는 건 고통스럽다. 지금 몇 단계인지, 다음에 뭘 넣는지 한눈에 안 들어온다.

이 앱은 이 세 가지를 **재료(일본 슈퍼 한정) → 양(인분 자동 계산) → 조리(모바일 단계 뷰)** 흐름으로 한 번에 해결한다.

### 2.2 타겟 사용자 & 사용 시나리오

**주 페르소나 — "도쿄 자취러 taemin"**
- 도쿄 거주 한국인 직장인. 한국 음식이 그립지만 매번 외식/배달은 부담.
- 요리 초·중급. 레시피만 정확하면 따라 할 수 있으나, 재료 대체와 분량 조절에서 막힌다.
- 근처 슈퍼(라이프, 마이바스켓, 세이유, 업무슈퍼 등)에서 장을 본다.
- 폰으로 요리하고, 앱 설치 없이 링크만으로 바로 쓰길 원한다.

**대표 사용 상황**
1. **퇴근 후 혼밥 (평일 저녁):** 배고픈 상태로 귀가. 냉장고 + 슈퍼에서 급히 살 것으로 30분 내 한 그릇. → 한식 → 김치볶음밥 → 1인분 → 재료·양 확인 후 바로 조리 뷰.
2. **주말 손님상 (4~5인분):** 친구 초대해 소고기미역국 + 반찬. → 메뉴 선택 → 4인분 입력 → 장보기 리스트가 4인분 기준 자동 산출 → 슈퍼에서 그대로 구매 → 집에서 단계 가이드로 조리.
3. **장보기 전 계획 (슈퍼 가는 길):** 오늘 뭐 해 먹지? → 카테고리 훑으며 메뉴 결정 → 인분 설정 → 재료 목록을 장보기 체크리스트로 사용(한/일 병기라 매대에서 바로 찾음).

### 2.3 핵심 가치 제안

일반 레시피 블로그/유튜브 대비 차별점 3가지.
1. **인분 자동 계량:** 인분 수만 입력하면 물 ml, 간장 큰술, 소금 g까지 전 재료가 특성에 맞게 재계산. 머릿속 비율 계산·환산 실수 제로.
2. **일본 슈퍼 재료 한정 + 한/일 병기:** 모든 레시피가 도쿄 슈퍼에서 실제로 살 수 있는 재료로만 구성. 재료명을 한국어·일본어로 함께 표기해 매대에서 헤매지 않는다.
3. **모바일 조리 전용 뷰:** 광고·잡담 없이 재료 → 단계만. 큰 글씨, 한 번에 한 단계, 젖은 손으로도 넘기기 쉬운 UI.

### 2.4 기능 목록 & MVP 범위

우선순위: **P0 = MVP 필수(첫 출시)**, **P1 = 다음 반복**, **P2 = 여유되면**.

| 기능 | 우선순위 | 근거 |
|---|---|---|
| 카테고리(한/양/중/일) → 메뉴 리스트 탐색 | P0 | 앱의 뼈대 |
| 메뉴 상세: 재료 목록 + 정확한 양 | P0 | 핵심 가치의 출발점 |
| 인분 수 입력 → 재료 양 자동 계산 | P0 | 최대 차별점 |
| 단계별 조리 가이드 (모바일 뷰) | P0 | 실사용 완결 |
| 재료명 한/일 병기 | P0 | 일본 슈퍼 사용성의 핵심 |
| 모바일 우선 반응형 레이아웃 | P0 | 폰에서 요리하며 보는 게 전제 |
| GitHub Pages 정적 배포 | P0 | 백엔드 없이 무료 배포 |
| 초기 시드 레시피 10~15개 | P0 | 실사용 최소 콘텐츠 |
| 메뉴 검색 / 재료로 필터 | P1 | 레시피 수 늘면 필요 |
| 장보기 체크리스트(재료 체크·복사) | P1 | 편의 기능 |
| 즐겨찾기 / 최근 본 메뉴 (localStorage) | P1 | 반복 사용 편의 |
| 조리 타이머 (단계 내 시간 버튼) | P1 | 폰 기본 타이머로 대체 가능 |
| 대체재 안내 (재료별 "없으면 이걸로") | P2 | 콘텐츠 성숙 후 |
| 재료 단위 환산(큰술↔ml) 토글 | P2 | 니치 편의 |
| PWA(오프라인·홈 화면 추가) | P2 | 정적앱이라 나중에 얹기 쉬움 |
| 사진/영상 스텝 이미지 | P2 | 제작 비용 큼, 텍스트로 시작 |

**MVP 정의:** P0만으로 "카테고리에서 메뉴 고르고 → 인분 정하면 재료·양이 딱 나오고 → 단계 따라 요리 완성"이 되는 상태. 레시피는 JSON으로 관리해 코드 수정 없이 추가.

### 2.5 사용자 스토리
1. 도쿄 자취러로서, **카테고리에서 오늘 먹고 싶은 요리를 빠르게 고르**고 싶다, 왜냐하면 매번 뭘 해 먹을지 정하는 것부터가 일이기 때문이다.
2. 혼밥하는 사람으로서, **인분을 1인분으로 바꾸면 재료 양이 알아서 줄어들**길 원한다, 왜냐하면 2인분 레시피를 반으로 나누다 자주 틀리기 때문이다.
3. 손님을 초대한 사람으로서, **4인분 기준 재료 양을 한 번에 보**고 싶다, 왜냐하면 여러 명 분량을 머릿속으로 곱하기 싫기 때문이다.
4. 일본 슈퍼에서 장 보는 사람으로서, **재료명을 일본어로도 보**고 싶다, 왜냐하면 매대에서 한국어 이름으로는 못 찾기 때문이다.
5. 요리 초보로서, **정확한 양(물 500ml, 간장 2큰술)을 애매한 표현 없이** 알고 싶다, 왜냐하면 "적당히"가 제일 어렵기 때문이다.
6. 폰으로 요리하는 사람으로서, **한 화면에 한 조리 단계씩 큰 글씨로** 보고 싶다, 왜냐하면 젖은 손으로 작은 글씨를 스크롤하기 힘들기 때문이다.
7. 장 보러 가는 사람으로서, **필요한 재료 목록을 체크리스트처럼** 쓰고 싶다, 왜냐하면 빠뜨리는 재료 없이 한 번에 사고 싶기 때문이다. (P1)
8. 자주 해 먹는 사람으로서, **즐겨 찾는 메뉴를 저장**해두고 싶다, 왜냐하면 매번 카테고리를 뒤지기 번거롭기 때문이다. (P1)

### 2.6 성공 지표 (개인용 기준)
**정량**
- 주 1회 이상 실제 요리에 앱 사용.
- 초기 시드 레시피 15개 확보, 이후 월 2~3개씩 추가.
- 인분 자동 계산 오류 0건.

**정성**
- "레시피 검색 → 대체재 찾기 → 분량 환산"에 쓰던 시간이 체감상 사라졌는가.
- 슈퍼에서 재료를 헤매지 않고 바로 담았는가.
- 이 앱만 보고 요리를 끝까지 완성했는가(중간에 다른 블로그를 안 켰는가).

### 2.7 로드맵
- **Phase 1 — MVP (첫 출시):** P0 전부 + 시드 레시피 10~15개. 목표: 평일 저녁 혼밥에 실제로 쓰기 시작.
- **Phase 2 — 실사용 편의:** 검색·필터, 장보기 체크리스트, 즐겨찾기·최근 본 메뉴, 조리 타이머. 레시피 30~50개로 확대(양·중식 채우기).
- **Phase 3 — 완성도·콘텐츠 성숙:** 대체재 안내, 단위 환산 토글, PWA(오프라인/홈화면), 단계별 사진(선택). 목표: 인터넷 없이도 대응하는 "내 전용 요리책".

### 3.1 일본 슈퍼 식재료 가이드

도쿄 거주자가 라이프(ライフ)·서밋(サミット)·이토요카도(イトーヨーカドー)·마이바스켓(まいばすけっと)·업무슈퍼(業務スーパー)에서 조달하는 것을 전제로 한다. 한식 재료 중 일부는 일반 슈퍼에 없고 **업무슈퍼**나 **한인마트(신오쿠보 韓国広場·韓国食品館 등)**에서 사야 하므로 열에 명시했다.

**가장 헷갈리는 포인트 먼저:**
- **국간장 vs 진간장**: 한국 진간장 ≈ 濃口醤油(진한간장), 한국 국간장은 색이 옅은 薄口醤油가 가장 가깝다. 단 薄口는 오히려 **염도가 더 높으니** 국간장 레시피에 쓸 때는 양을 살짝 줄이고 간을 본다. 발효 감칠맛은 약하므로 국물용은 だしの素·소금으로 보완.
- **고춧가루**: 일반 슈퍼의 一味唐辛子는 너무 곱고 매워 부적합. 반드시 **업무슈퍼/한인마트의 굵은 韓国唐辛子(粉唐辛子)**.
- **다시다 대체**: 소고기맛=味覇(ウェイパー)·牛だし, 범용 감칠맛=鶏がらスープの素, 국물 베이스=だしの素. 다시다 자체는 업무슈퍼·한인마트에서 ダシダ로 구입 가능.
- **미역**: 국용은 큰 조각 필요. 일반 슈퍼의 カットわかめ는 잘게 잘려 미소시루용. 업무슈퍼의 큰 乾燥わかめ(韓国産·鳴門産) 권장.

### 3.2 계량 단위 & 스케일링 (도메인 근거)

계량 표준과 `scaleType` 5종·표시 반올림 규칙은 **§1.3~§1.4에 확정본**으로 정리되어 있다. 도메인 관점의 근거만 보강한다.

- **`sqrt`(간·향)의 근거:** 간과 향은 인분에 정비례로 늘리면 짜거나 매워진다(부피/미각 인지 비선형). 실전 가정요리에서 검증된 절충이 제곱근 비례. 예) 2인분 소금 1작은술 → 8인분: linear면 4작은술(과함), √4=2배인 2작은술이 현실적.
- **`count`(계란·두부 등):** 2→3인분에서 계란 2개×1.5=3개(정수, 문제없음). 소수가 남으면 roundStep(기본 1, 반 단위 재료는 0.5)로 반올림하고 "0.5개는 풀어서 절반 사용" 비고 자동 노출. 두부·양파·대파는 roundStep 0.5 권장.
- **비례하지 않는 것:** 조리시간은 `fixed`(예열·끓는 시간은 정비례 아님). 물 증발량은 `linear`로 두되 `ratio≥3`이면 "물 5~10% 감량" 안내. 부침/코팅 기름은 `fixed` 또는 약한 스케일.

### 3.3 레시피 데이터 모델 (필드 설명)

전체 스키마는 **§1.5 확정본** 참조. 도메인 관점 필드 설명:

| 필드 | 설명 |
|---|---|

| `baseServings` | 기준 인분(대개 2) |

| `difficulty` | easy·normal·hard |
| `tags` | 예: 국물·매운맛·한그릇·10분 |

### 3.4 샘플 레시피 3건 (완전판, 기준 2인분)

재료 표는 그대로 JSON `ingredients[]`로 옮길 수 있게 구조화했다.

#### 3.4.1 소고기미역국 (牛肉わかめスープ) — 한식 · easy · 약 35분

| 재료(한/일) | 기준량(2인분) | 단위 | scaleType | 비고 |
|---|---|---|---|---|
| 마른 미역 (乾燥わかめ) | 10 | g | linear | 불리면 5~6배, 큰 조각 |
| 소고기 국거리·양지 (牛肉 煮込み用/牛バラ切り落とし) | 150 | g | linear | 한입 크기 |
| 참기름 (ごま油) | 1 | 큰술 | linear | 볶음용 |
| 국간장 (薄口醤油) | 1 | 큰술 | linear | 薄口는 짜니 간 보며 |
| 다진 마늘 (おろしにんにく) | 1 | 작은술 | sqrt | |
| 물 (水) | 1000 | ml | linear | ratio≥3이면 5~10% 감량 |
| 국물용 다시 (だしの素/牛だし) | 1 | 작은술 | linear | optional, 감칠맛 |
| 소금 (塩) | null | 적당량 | to-taste | 마지막 간 |

**조리 단계**
1. 마른 미역을 찬물에 15분 불린 뒤 두세 번 헹궈 물기를 꼭 짜고 먹기 좋게 썬다. (5분)
2. 냄비에 참기름을 두르고 소고기를 중불에 볶아 겉면이 익으면 미역을 넣고 2~3분 함께 볶는다. (5분)
3. 국간장을 넣어 30초 볶아 향을 낸 뒤 물·다진 마늘·다시를 넣는다. (2분)
4. 끓어오르면 거품을 걷고 중약불로 20~25분 끓인다. (23분)
5. 소금으로 최종 간을 맞춘다.

**팁:** 참기름에 미역을 충분히 볶아야 국물이 뽀얗고 고소하다. 소고기 대신 조갯살(あさり)도 가능.

#### 3.4.2 김치볶음밥 (キムチチャーハン) — 한식 · easy · 약 15분

|---|---|---|---|---|
| 신 김치 (酸っぱいキムチ) | 200 | g | linear | 잘 익은 것 |
| 밥·찬밥 (ご飯/冷やご飯) | 400 | g | linear | 2공기, 찬밥이 덜 질음 |
| 대파 (長ねぎ) | 0.5 | 대 | count | roundStep 0.5, 송송 |
| 식용유 (サラダ油) | 1 | 큰술 | linear | 파기름용 |
| 참기름 (ごま油) | 1 | 작은술 | linear | 마무리 |
| 고추장 (コチュジャン) | 1 | 큰술 | sqrt | 매움 조절 |
| 설탕 (砂糖) | 1 | 작은술 | linear | 신맛 완화 |
| 간장 (醤油) | 1 | 작은술 | linear | |
| 계란 (卵) | 2 | 개 | count | roundStep 0.5, 프라이 |
| 스팸·돼지고기 (ランチョンミート/豚肉) | 100 | g | linear | optional |
| 구운 김·김가루 (韓国のり/きざみのり) | null | 적당량 | to-taste | 고명 |

**조리 단계**
1. 김치는 잘게 썰고 대파는 송송 썬다. 스팸을 쓰면 사방 1cm로 썬다. (3분)
2. 팬에 식용유를 두르고 대파를 중불에 볶아 파기름을 낸다. (2분)
3. (스팸·고기 사용 시) 넣어 노릇하게 볶는다. (2분)
4. 김치를 넣고 설탕을 뿌려 3~4분 볶아 신맛을 죽이고 숨을 죽인다. (4분)
5. 밥을 넣고 고추장·간장을 넣어 덩어리 없이 골고루 볶는다. (3분)
6. 불을 끄고 참기름·통깨를 두른다.
7. 계란 프라이를 부쳐 위에 올리고 김가루를 뿌린다.

**팁:** 신맛 부족하면 식초 몇 방울. 고추장 대신 고춧가루 1큰술이면 더 개운.

#### 3.4.3 오야코동 (親子丼) — 일식 · easy · 약 20분

일본 슈퍼 재료(みりん·だしの素)만으로 완성되는 대표 한그릇.

| 재료(한/일) | 기준량(2인분) | 단위 | scaleType | 비고·group |
|---|---|---|---|---|
| 닭다리살 (鶏もも肉) | 250 | g | linear | 한입 크기 |
| 양파 (玉ねぎ) | 0.5 | 개 | count | roundStep 0.5, 채썰기 |
| 계란 (卵) | 4 | 개 | count | roundStep 0.5, 살짝 풀기 |
| 따뜻한 밥 (ご飯) | 400 | g | linear | 2공기 |
| 물 (水) | 150 | ml | linear | group: 조미액 |
| 간장 (醤油) | 2 | 큰술 | linear | group: 조미액 |
| 미림 (みりん) | 2 | 큰술 | linear | group: 조미액 |
| 설탕 (砂糖) | 1 | 큰술 | linear | group: 조미액 |
| 다시 (だしの素) | 1 | 작은술 | linear | group: 조미액 |
| 파드득나물·쪽파 (三つ葉/青ねぎ) | null | 적당량 | to-taste | 고명 |

**조리 단계**
1. 닭다리살은 한입 크기, 양파는 채썬다. 계란은 흰자가 살짝 남을 정도로만 가볍게 푼다. (5분)
2. 작은 팬에 조미액(물·간장·미림·설탕·다시)을 넣고 끓인다. (2분)
3. 양파를 넣어 2~3분 익힌다. (3분)
4. 닭고기를 넣고 중불에서 5~6분, 속까지 익힌다. (6분)
5. 풀어둔 계란의 2/3를 원을 그리며 붓고, 반쯤 익으면 나머지 1/3을 붓는다. 뚜껑 덮고 30초~1분 뜸 들여 반숙. (2분)
6. 밥 위에 국물째 얹고 파드득나물·쪽파를 올린다.

**팁:** 계란은 완전히 익히지 말고 촉촉할 때 불을 끈다. 닭 대신 돼지고기면 他人丼(타닌동).

## 4. UX/UI 설계

### 4.1 정보 구조(IA) & 화면 목록

얕고 명확한 **3-depth 구조**. 조리 중 길을 잃지 않도록 계층 최소화, 언제든 홈/카테고리로 한 번에 복귀.

```
[홈 / 카테고리 선택]  (Depth 0, 진입점)
│
├── 한식 ─┐
├── 양식 ─┤
├── 중식 ─┼──▶ [메뉴 리스트] (Depth 1, 카테고리별)
└── 일식 ─┘        │
                    └──▶ [레시피 상세] (Depth 2)
                             ├─ 인분 스테퍼
                             ├─ 재료 목록(자동 계산)
                             └─ 조리 단계 가이드
[전역 진입]
├── [검색]     (모든 화면 상단)
└── [즐겨찾기]  (탭바/홈)
```

| 화면 | ID | 목적 |
|---|---|---|
| 홈/카테고리 | `home` | 4개 카테고리 그리드 진입점, 검색·즐겨찾기 접근 |
| 메뉴 리스트 | `menu-list/:category` | 카테고리의 요리 목록. 썸네일·이름·시간·난이도 |
| 레시피 상세 | `recipe/:id` | 핵심 화면. 인분 조절 → 재료 자동 계산 → 단계 가이드 |
| 검색 | `search` | 요리명·재료명(한/일) 교차 탐색 (P1) |
| 즐겨찾기 | `favorites` | 북마크 모음(localStorage) (P1) |

- 정적 SPA, 각 화면 딥링크 가능(URL 공유·북마크). 인분 수는 URL 쿼리(`?n=2`)에 반영해 새로고침·공유 시 유지.

### 4.2 핵심 사용자 플로우

```
[홈] ─ 카테고리 카드 탭(예: 한식)
  ▼
[메뉴 리스트·한식] ─ 메뉴 카드 탭(예: 김치찌개)
  ▼
[레시피 상세]
  ├─ 진입 시 기본 인분: 2 (또는 마지막 선택값)
  ├─ [- 2인분 +] 스테퍼 → 재료량 실시간 재계산(화면 이동 없음)
  ├─ 재료 목록 확인(한/일 병기)
  └─ 조리 단계 스크롤 + 스텝 완료 체크
  ▼
[완료] 마지막 스텝 체크 시 완료 축하 상태
```

**내비게이션 규칙**
- 뒤로가기: 상단 좌측 `‹ 뒤로`는 직전 화면으로. 브라우저 back 제스처와 동일.
- 홈 복귀: 하단 탭바 홈 아이콘으로 어느 depth에서든 즉시 복귀.
- 카테고리 전환: 메뉴 리스트 상단 **카테고리 탭(한/양/중/일)** 항상 노출 → 뒤로 안 가고 옆 카테고리로 전환.
- 상태 보존: 상세 → 뒤로 → 재진입 시 인분 선택값·스텝 체크 유지.
- 하단 탭바: 홈·검색·즐겨찾기 3탭. 레시피 상세에서 조리 몰입 위해 스크롤 시 자동 숨김.

### 4.4 인분 조절 UX
- **스테퍼 조작 → 즉시 재계산:** `+/−` 탭 시 화면 이동·리로드 없이 재료 수량이 짧은 하이라이트와 함께 갱신. "재료 (N인분 기준)" 라벨 동시 갱신.
- **계산·표기:** 표시값 = §1.4 규칙(scaleType별 계산 + 표시 반올림 + 분수 표기). 분수 매핑 `¼ ⅓ ½ ⅔ ¾`, 정수부 앞에(`1½`). 유니코드 분수 글리프, 폴백 `1/2`.
- **범위:** UI 스테퍼 1~4. 경계에서 버튼 비활성(회색). 기본값 최초 2인분, 마지막 선택값 로컬 저장(선택).

### 4.5 조리 중 사용성 (Cooking Mode)
젖은 손·바쁜 상황 전제.
1. **화면 꺼짐 방지(Wake Lock):** 상세 진입 시 Screen Wake Lock, 명시 토글(기본 ON). 미지원 브라우저는 토글 숨김.
2. **큰 글씨·넉넉한 행간:** 조리 단계 본문 최소 17px(조리모드 19px+), 줄간격 1.6+. 재료 수량 굵게.
3. **큰 탭 타겟:** 모든 상호작용 요소 최소 48×48px.
4. **스텝 완료 토글:** 단계 탭 시 완료 체크(취소선/회색), 진행률(2/5) 표시.
5. **스크롤 최소화:** 재료↔조리 점프 앵커, 현재 스텝 자동 하이라이트.
6. **오조작 방지:** 파괴적이지 않은 액션만 상단, 화면 벗어나도 상태 보존.
7. **다크·저조도 배려, 오프라인 견고성(서비스워커 캐시, 선택).**

| 유지보수(비개발자) | Node·의존성 불필요 → **부담 최소** | node_modules·버전·취약점 관리 |
| 컴포넌트 재사용/확장성 | 수동(ES 모듈) | 우수 |
| 번들/성능 | 매우 가벼움(0KB) | 런타임 40KB+ |

**추천: 옵션 A (바닐라 + ES 모듈).** 이 앱의 복잡성은 UI가 아니라 **데이터/계산 로직**에 있고, 그 부분은 프레임워크와 무관하게 순수 JS 모듈로 잘 분리된다. **빌드가 없다**는 점이 개인 프로젝트에서 결정적 — `git push` 한 번으로 배포 끝, 의존성 취약점·재빌드 부채 0, 비개발자도 웹 에디터에서 JSON만 고치면 즉시 반영. 화면이 크게 복잡해지면 그때 React로 옮기되, **데이터 스키마·계산 엔진은 그대로 재사용**되어 전환 비용이 낮다.

**PWA:** 두 옵션 다 가능, **Phase 2 선택**. `manifest.webmanifest` + 간단한 서비스워커(app shell·데이터 cache-first)만 추가하면 오프라인 열람·홈화면 추가.

### 5.3 인분 계산 엔진 설계

`ratio = target / baseServings`를 구하고, 재료마다 `scaleType`에 따라 스케일 → 단위별 반올림 → 사람이 읽는 표기(분수 포함)로 변환.

```js
// js/scaler.js — 순수 함수, 부수효과 없음
export function scaleAmount(amount, scaleType, ratio, roundStep = 1) {
  switch (scaleType) {
    case 'linear':   return amount * ratio;
    case 'sqrt':     return amount * Math.sqrt(ratio);
    case 'fixed':    return amount;
    case 'count':    return Math.max(roundStep, Math.round(amount * ratio / roundStep) * roundStep);
    case 'to-taste': return null;                 // 계산 제외
    default:         return amount * ratio;        // 안전 폴백 = linear
  }
}

export function scaleIngredient(ing, ratio) {
  if (ing.scaleType === 'to-taste' || ing.amount == null)
    return { ...ing, value: null, display: '기호에 따라' };
  const raw = scaleAmount(ing.amount, ing.scaleType, ratio, ing.roundStep ?? 1);
  const value = ing.scaleType === 'count' ? raw : roundByUnit(raw, ing.unit);
  return { ...ing, value, display: formatAmount(value, ing.unit) };
}

export function scaleRecipe(recipe, targetServings) {
  const ratio = targetServings / recipe.baseServings;
  return { ...recipe, targetServings,
    ingredients: recipe.ingredients.map(ing => scaleIngredient(ing, ratio)) };
}
```

```js
// js/format.js — 단위별 반올림 + 분수 표기
const UNIT_STEP = { ml: 5, g: 5, '큰술': 0.5, '컵': 0.5, '작은술': 0.25, '개': 1, '모': 0.5, '대': 0.5, '쪽': 1 };
export function roundByUnit(amount, unit) {
  const step = UNIT_STEP[unit];
  if (!step) return amount;                        // 적당량 등 그대로
  return +(Math.round(amount / step) * step).toFixed(2);
}

const FRACTIONS = [[0,''],[0.25,'¼'],[0.33,'⅓'],[0.5,'½'],[0.67,'⅔'],[0.75,'¾'],[1,'']];
export function toFraction(value) {
  let whole = Math.floor(value); const frac = value - whole;
  let best = FRACTIONS[0];
  for (const f of FRACTIONS) if (Math.abs(frac - f[0]) < Math.abs(frac - best[0])) best = f;
  if (best[0] === 1) whole += 1;
  const glyph = best[0] === 1 ? '' : best[1];
  if (!glyph) return String(whole);
  return whole === 0 ? glyph : `${whole}${glyph}`;
}
export function formatAmount(value, unit) {
  if (value == null || unit === '적당량' || value === 0) return '기호에 따라';
  const num = Number.isInteger(value) ? String(value) : toFraction(value);
  return `${num} ${unit}`;
}
```

| 재료 | scaleType | 원값 | raw | 표시 |
|---|---|---|---|---|
| 마른 미역 | linear | 10 g | 25 | `25 g` |
| 소고기 | linear | 150 g | 375 | `375 g` |
| 물 | linear | 1000 ml | 2500 | `2500 ml` (안내: 2300ml로 시작) |
| 국간장 | linear | 1 큰술 | 2.5 | `2½ 큰술` |
| 다진 마늘 | sqrt | 1 작은술 | 1.58 | `1½ 작은술` |
| 소금 | to-taste | null | — | `기호에 따라` |

### 6.2 인분 스케일링 테스트 케이스

계산 규칙: `r = 목표/기준`. linear=`양×r`, sqrt=`양×√r`, fixed=`양`, count=`round(양×r/step)×step, 최소 step`.

|---|---|---|---|---|---|---|---|---|---|
| 1 | 정수 배수 | 물 | linear | 2 | 200ml | 4 | 2 | 400 | `400ml` |
| 2 | 축소 | 물 | linear | 2 | 200ml | 1 | 0.5 | 100 | `100ml` |
| 3 | 비정수(2→3) | 물 | linear | 2 | 200ml | 3 | 1.5 | 300 | `300ml` |
| 4 | 홀수 결과 | 간장 | linear | 2 | 30ml | 3 | 1.5 | 45 | `45ml` |
| 5 | count 딱맞음 | 계란 | count | 2 | 2개 | 3 | 1.5 | 3 | `3개` |
| 6 | count 올림 | 계란 | count | 2 | 1개 | 3 | 1.5 | 2 | `2개` |
| 7 | count 내림 | 계란 | count | 4 | 3개 | 3 | 0.75 | 2 | `2개` |
| 8 | sqrt 확대 | 소금 | sqrt | 2 | 4g | 4 | 2 | 5.657 | `5.7g`(스냅 후 5g/5.5g) |
| 9 | sqrt 축소 | 소금 | sqrt | 4 | 8g | 2 | 0.5 | 5.657 | `5.7g` |

| R05 | baseServings | 양수(정수 또는 0.5 배수) |
| R06 | ingredients | 최소 1개 |
| R07 | 재료 한/일 | 각 재료 `name.ko`, `name.ja` 채움 |
| R08 | amount | 숫자 ≥ 0, 또는 to-taste면 null |
| R09 | unit | 화이트리스트(§1.3) 내 |
| R10 | scaleType | `linear\|sqrt\|count\|fixed\|to-taste` |
| R11 | count 정합성 | scaleType=count면 unit은 개수 단위(개·모·대·장·쪽) |
| R12 | steps | 배열, 최소 1단계, 빈 항목 없음 |
| R13 | category | 한식·양식·중식·일식 중 하나, 파일명과 일치 |
| R14 | 스케일 항등 | baseServings→baseServings 재계산 시 원본과 동일 |
| R15 | 인코딩 | UTF-8, 한자/한글 깨짐 없음 |

검증 스크립트는 비제로 종료 + 위반 규칙 ID·재료 인덱스 출력(예: `R07 위반: ingredients[3].name.ja 누락`).

### 6.4 테스트 전략 & 도구
| 계층 | 대상 | 도구 | 범위 |
|---|---|---|---|
| 단위 | 계산 엔진(scaler·format) | **Vitest**(또는 `node --test`+assert) | §6.2 17케이스 + 경계값, 커버리지 100% |
| 데이터 검증 | 레시피 JSON | **ajv** 또는 R01~R15 커스텀 validator | 커밋 전 필수. `npm run validate:recipes` |
| 통합/스모크 | 렌더 + 인분 변경 UI | 대표 2~3개 수동/컴포넌트 | 핵심 플로우 |
| 수동 조리 | 실제 맛 | 대표 레시피를 기준/확대·축소 인분으로 실제 조리 | 신규 등록 시 1회, sqrt 재료 대량 검증 |
| 크로스브라우저/모바일 | 레이아웃·조작 | 본인 iPhone Safari + Android Chrome, DevTools 반응형 | 320px~ 실폭 |

### 6.5 수용 기준 (Acceptance Criteria)
**기능**
- [ ] 목록에서 레시피 선택해 상세를 연다.
- [ ] 상세에서 인분을 변경할 수 있다.
- [ ] 인분 변경 시 모든 재료 양이 scaleType에 맞게 즉시 갱신.
- [ ] 재료명이 한/일 병기로 표시.
- [ ] 실제 레시피 데이터 5개 이상(MVP 목표 10~15개) 등록.

**계산 정확도**
- [ ] §6.2 17케이스 전부 통과.
- [ ] linear/sqrt/fixed/count/to-taste 모두 최소 1개 레시피에서 실동작 확인.
- [ ] 분수·특수 표기(½,¼,⅓,¾,1½)가 매핑대로 표시.
- [ ] 표시 반올림이 내부 정밀도를 훼손하지 않음(연쇄 계산 오차 누적 없음).

**모바일 사용성**
- [ ] 320px에서 레이아웃 안 깨짐.
- [ ] iOS Safari·Android Chrome 실기기 정상.
- [ ] 인분 변경 한 손 조작 가능.
- [ ] 긴 재료명/큰 인분(10L)에서도 값 안 잘림.

**데이터 품질**
- [ ] 등록 레시피 전부 R01~R15 통과.

### 6.6 엣지 케이스 & 예외 처리
| 상황 | 기대 동작 |
|---|---|
| 목표인분 0 | 계산 안 함. 최소 1(또는 0.5)로 클램프 또는 "인분 선택" 안내. NaN/0 출력 금지 |
| 목표인분 음수 | UI 감소 버튼이 하한 이하로 차단. 직접 입력 시 무시/클램프 |
| 매우 큰 인분 | 정상 계산 + 큰 단위 환산 표시(10L 등). Infinity 없음 |
| amount 0 | 0×배율=0 유지, "기호에 따라"로 표시, NaN 금지 |
| amount null/누락 | to-taste 처리 또는 검증(R08) 차단. 런타임 도달 시 스킵 + 경고, 크래시 금지 |
| scaleType 누락/오타 | 검증(R10) 차단. 런타임 폴백 `linear` + 경고 |
| baseServings 0 | 0 나누기 → 검증(R05) 차단. 런타임 도달 시 계산 중단 + 에러 표시 |
| 없는 레시피 URL/id | 404 대체 화면 또는 목록 리다이렉트. 흰 화면/크래시 금지 |
| 깨진 JSON | 빌드 시 검증 실패로 배포 차단. 런타임엔 해당 레시피만 숨김 |
| 부동소수 오차 | 표시 직전 자릿수 반올림, 테스트는 ±0.01 근사 비교 |
